# Terraform / DevOps — rules

Long-form companion to [skills/terraform-devops/SKILL.md](../skills/terraform-devops/SKILL.md).

## Repository layout

```
infra/
  envs/
    dev/        main.tf  variables.tf  terraform.tfvars  backend.tf
    staging/
    prod/
  modules/
    network/    main.tf  variables.tf  outputs.tf  versions.tf  README.md
    database/
    eks-cluster/
  policies/     # OPA / Sentinel policies
```

Why a directory per env, not workspaces? Workspaces share a state backend and provider config; that's the wrong blast radius. A dev mistake should not be able to reach prod state.

## Modules — when and how

Make a module when:

- The same shape appears in two or more environments and is non-trivial to wire up.
- A team or product needs a curated abstraction (e.g. "our standard EKS cluster").

Don't make a module when:

- You're just renaming `aws_s3_bucket` to `module.bucket`. That's a wrapper, not a module.
- You can't write a README explaining what it does in three sentences.

Module contract:

- Inputs are validated (`validation` blocks).
- Outputs are stable — renaming an output is a breaking change.
- Version with semver tags. Consumers pin to a tag.

## State

- One backend per env. Different bucket key, different lock table entry.
- State file contains secrets (DB passwords, private keys generated by TF). Treat it as a secret artifact: encrypted at rest, access-logged, versioned.
- Never `terraform state rm` to fix a bad apply unless you know exactly what you're removing and have a backup.
- `terraform import` for resources created outside TF. Document the import in a comment so the next person knows.

## Plans and applies

- Every change runs through CI: fmt, validate, plan, then a manual approve before apply.
- The plan output is the source of truth for what will change. Read it. "Plan looks right" is not a substitute for reading it.
- Refuse to apply a plan more than ~30 minutes old in a busy environment — state has likely drifted.

## Drift

Drift happens. Causes, in order of frequency:

1. Someone made a console change.
2. Auto-scaling / managed services updated something Terraform tracks (cluster version, ASG capacity).
3. A provider upgrade computed a default differently.

Responses:

- Console change → revert in console or import the change to code.
- Managed updates → use `ignore_changes` on the specific attribute (`desired_capacity`), not `all`.
- Provider upgrade → upgrade deliberately, read the changelog, run plan in dev first.

## IAM

- Roles per workload, not per person.
- Least privilege: start from zero, add what the plan/apply tells you is missing.
- Never inline policies that grant `*` on `*`. Even break-glass roles should restrict by service and region.
- Boundaries: SCPs at the org level, permission boundaries for delegated admin.

## Secrets

- Generated secrets (DB passwords, JWT signing keys) live in Secrets Manager / SSM. Terraform creates the secret and writes the value, but the value should be `random_password` from the provider or an external source — not a literal.
- Consumed secrets (third-party API keys) are created out-of-band; Terraform references them by ARN.
- Never `output` a secret unless it's marked `sensitive = true`. Even then, the value lands in state.

## Kubernetes / Helm

- Cluster created by Terraform; workloads by GitOps (Flux/Argo) reading a manifests repo. Don't mix.
- Helm values come from version-controlled files. No `--set` in `helm upgrade` for anything that should persist.
- A separate "platform" repo holds cluster add-ons (ingress, cert-manager, monitoring). Apps don't install their own.

## Cost

- Tag every resource: `Environment`, `Owner`, `CostCenter`, `Project`. Enforce via a default tags provider block, not by hand.
- Set up budget alerts in code (`aws_budgets_budget`), not in the console.
- Right-size before you commit. A `db.r6g.4xlarge` for "we might need it" is the most expensive bug you'll ship.

## Destroy

`terraform destroy` is a code review event. Treat the command like a `DROP DATABASE`:

- Run plan first.
- Require a written approval in the PR or ticket.
- For prod, two-person rule.
- Backups verified beforehand. "We have backups" ≠ "we tested the restore."

## What this skill is *not*

- Application deployment / CI/CD (use cicd).
- Cloud architecture decisions at the "should we use Kubernetes" level — those need a real design doc.
- Specific cloud SDK code (boto3, gcloud, etc.) inside applications.
